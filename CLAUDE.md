## Project Overview

A Dart FFI project implementing a cross-platform virtual router engine — a reverse tethering tool that bridges Android devices to internet connectivity via libslirp (user-space TCP/IP stack). Analogous to [gnirehtet](https://github.com/Genymobile/gnirehtet) but implemented in Dart using FFI.

## Commands

```bash
# Run the main router application
dart run bin/gnirehtet.dart

# Run the basic FFI demo (processor.c example)
dart run main.dart

# Regenerate FFI bindings from C headers (when libslirp headers change)
dart run ffigen

# Build C libraries manually (see .vscode/tasks.json for reference)
clang -shared -o libprocessor.dylib processor.c
```

## Architecture

### Three-Layer Network Bridge

```
Android Device (USB)
    ↕  AdbManager — reverse port forwarding via `adb reverse`
TunnelServer — localhost TCP socket (port 31416), relays raw IP packets
    ↕  RouterEngine — orchestrates all subsystems
SlirpWrapper — Dart FFI bridge to libslirp
    ↕  libslirp.dylib — user-space NAT/TCP/IP stack (C library)
    ↕  Host network interfaces
```

### Key Components

**`lib/src/core/router_engine.dart`** — Central orchestrator. Manages lifecycle, coordinates data flow between ADB tunnel ↔ libslirp ↔ network.

**`lib/src/ffi/slirp_wrapper.dart`** — Core FFI complexity lives here. Manages:
- 7 C callback registrations (send packet, poll sockets, timers)
- A 10ms polling loop (`slirp_pollfds_fill` → `slirp_pollfds_poll`)
- Ethernet frame wrapping/stripping (libslirp speaks Ethernet; the tunnel speaks raw IP)
- ARP spoofing to satisfy libslirp's ARP requirement

**`lib/src/ffi/slirp_bindings.dart`** — Auto-generated by `ffigen` from libslirp headers. Do not edit manually.

**`lib/src/ffi/lib_loader.dart`** — Cross-platform dynamic library discovery (`.dylib`/`.dll`/`.so`).

**`lib/src/network/tunnel_server.dart`** — TCP server accepting a single device connection.

**`lib/src/network/adb_manager.dart`** — ADB environment check and reverse tunnel setup.

### Data Flow

```
Device → TunnelServer → RouterEngine → SlirpWrapper.inputFromDevice()
  → slirp_input() [C] → _cSendPacketCallback() [Dart static]
  → RouterEngine → TunnelServer → Device
```

### Critical FFI Patterns

- **Static callbacks**: `Pointer.fromFunction<>()` exposes Dart static methods to C
- **Opaque pointer trick**: Instance ID stored in C `opaque` pointer to retrieve the correct Dart wrapper instance inside callbacks
- **Persistent allocations**: `calloc` for C structs that must outlive the call (freed only in `dispose()`); `malloc` for temporary packet buffers (freed immediately)
- **Blind event polling**: `_blindEvents` list fakes socket readiness events to work around OS differences

### Ethernet Frame Handling

libslirp operates at Layer 2 (Ethernet), but the Android tunnel sends raw Layer 3 IP packets. `SlirpWrapper` adds/removes a 14-byte Ethernet header on every packet. EtherType is set dynamically based on the IP version byte (`0x0800` for IPv4, `0x86DD` for IPv6).

## FFI Binding Regeneration

`ffigen.yaml` configures auto-generation. When regenerating:
1. Ensure libslirp headers are available at the configured path
2. Run `dart run ffigen`
3. Output goes to `lib/src/ffi/slirp_bindings.dart`
